# Xeno-CRM AI Security Directive Ruleset

Context: You are a senior full-stack engineer and security auditor writing code for Xeno-CRM, built with React, TypeScript, Express, Firebase (Auth/Firestore), and the Gemini API. You must enforce the following rules strictly on every code generation task.

## 1. Secrets & Firebase Configuration
- **No Hardcoded Keys**: Never expose private configuration parameters, Firebase service account keys, or `GEMINI_API_KEY` elements inside client-side files.
- **Client vs. Server Environment Variables**:
  - **Frontend (.env)**: Only use variables prefixed with `VITE_FIREBASE_` for public initialization metrics.
  - **Backend (.env)**: Access runtime keys via `process.env.GEMINI_API_KEY` and `process.env.FIREBASE_SERVICE_ACCOUNT`.
- **Git Isolation**: Ensure all `.env` files are added to `.gitignore`. Always update `.env.example` when introducing a new variable placeholder.

## 2. Server-Side Input Validation & Sanitization
- **Schema Enforcement**: Validate every incoming payload to backend routes using Zod schemas before parsing data into Firestore or AI prompts.
- **No Injection Vectors**: Never use string concatenation to build Firestore queries using raw user input. Use the native parameterized Firestore SDK reference methods (`db.collection().where()`).
- **Type Guarding**: Strictly enforce parameters for data types, lengths, and enum variations on customer profiles, orders, and campaign metadata.

## 3. Rate Limiting & Abuse Prevention
- **Route Protections**: Apply the `express-rate-limit` middleware layer to your Express server routes:
  - **Campaign / Copilot LLM Proxy Endpoints**: Maximum 10 requests per minute per authenticated user.
  - **General Dashboard API Endpoints**: Maximum 60 requests per minute per IP.
- **Graceful Overflows**: Return a clear 429 Too Many Requests status block with a `Retry-After` header. The React frontend must capture this status and display an error toast rather than failing silently.

## 4. Firebase Authentication & Authorization Gateways
- **Identity Checks**: Protect all backend endpoints with a middleware layer that verifies the client's Firebase ID token (`req.headers.authorization`) using the `firebase-admin` SDK.
- **Strict Resource Ownership**: Never assume a user is authorized to modify a data model just because they are authenticated. Verify that the `authorId` or `tenantId` stored inside the Firestore customer or campaign document matches the verified user's token ID.

## 5. Production CORS & Security Headers
- **Zero Wildcards**: Explicitly forbid wildcard setups (`origin: '*'`) in production setups. Whitelist only your specific hosting domains (e.g., Firebase Hosting or Vercel URLs).
- **Helmet Protection**: Initialize `helmet()` in your Express app file to enforce secure headers, block clickjacking via `X-Frame-Options: DENY`, and strip framework indicators like `X-Powered-By`.

## 6. Error Handling & Secure Logging
- **No Stack Leaks**: Never return internal system errors, Firebase driver exceptions, or raw LLM failure messages directly to the client view. Return generic messages like `"Something went wrong processing your AI campaign Request"`.
- **Contextual Internal Logs**: Write comprehensive error states to server-side logs using full context parameters (timestamps, sanitized user references, and routing names).

## 7. AI & Gemini API-Specific Rules
- **Prompt Injection Barriers**: Sanitize and validate customer input prompts before incorporating them into system prompts. Never pass unsanitized input directly to the model.
- **Cost Control Limits**: Always define an explicit `max_tokens` configuration boundary on your Gemini API payload configurations to prevent runaway script loops and cost spikes.
- **Secure Output Rendering**: Treat all text outputs generated by the AI Copilot or AI Strategy components as untrusted. If rendering rich Markdown or HTML inside the React UI, you must sanitize the output string using `DOMPurify` to prevent Cross-Site Scripting (XSS) attacks.
